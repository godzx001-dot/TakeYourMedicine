# Fullstack Collab

## 适用场景

当用户需求同时涉及前端与后端时使用本 Skill：
- 新功能从 0 到 1（页面 + 接口 + 数据）
- 既要前端实现，也要 Node.js 后端实现
- 需要先定义接口契约，再并行开发与联调
- 希望交付文档、代码、测试与验收清单

## 协作目标

1. 让前后端围绕同一份接口契约协作，减少反复沟通
2. 用数据库设计保障后端实现正确性
3. 用联调与验收清单确保端到端可用

## 子 Skill 分工

- `frontend-dev`：
  - 页面信息架构、交互实现、状态管理
  - 输出接口需求文档（Frontend -> Backend）
- `backend-dev`：
  - 按接口需求 + 数据库设计实现 Node.js 服务
  - 输出正式 API 文档并支持联调

## 标准执行流程

### 阶段 1：需求对齐
- 明确业务目标、用户路径、优先级（P0/P1/P2）
- 确认范围：本次迭代页面、接口、数据变更
- 产出：需求摘要 + 范围边界 + 风险点

### 阶段 2：接口契约先行（由 frontend-dev 主导）
- 先建页面数据模型，再反推接口清单
- 输出接口需求文档：Method/Path、参数、响应、错误码、前端处理策略
- 标记优先级与依赖关系
- 产出：可执行接口需求文档 v1

### 阶段 3：后端设计与实现（由 backend-dev 主导）
- 对齐数据库表结构与接口字段映射
- 设计并实现 Router/Controller/Service/DAO
- 加入鉴权、校验、幂等、日志与 traceId
- 产出：可运行后端代码 + 正式 API 文档 v1

### 阶段 4：前端联调实现（frontend-dev）
- 对接真实接口，替换 mock
- 完成 loading/empty/error/retry 全状态处理
- 按错误码协议完成提示与降级
- 产出：联调通过页面与交互

### 阶段 5：联合验收与发布准备
- 按验收清单回归正常/异常/边界场景
- 检查文档与实现一致性（契约一致）
- 准备发布项：配置、迁移、回滚、监控告警
- 产出：验收报告 + 发布清单

## 交付物模板

### 1) 协作计划
- 里程碑（M1 接口冻结 / M2 联调完成 / M3 验收）
- 分工表（前端/后端）
- 风险与应对

### 2) 接口清单（最少字段）
- 接口名称
- Method + Path
- 输入参数与约束
- 输出结构与错误码
- 负责人
- 计划完成时间

### 3) 联调清单
- [ ] Token 鉴权通过
- [ ] 分页/筛选/排序正确
- [ ] 错误码处理一致
- [ ] traceId 可追踪
- [ ] 性能满足目标

### 4) 发布清单
- [ ] 环境变量确认
- [ ] DB 迁移脚本执行
- [ ] 回滚方案可用
- [ ] 监控与日志可观察

## 编排规则

1. 先契约后实现：接口未冻结前，不做大规模并行开发
2. 文档与代码同源更新：契约变更需同步更新两端文档
3. 小步快跑：按 P0 先闭环，再扩展 P1/P2
4. 问题闭环：联调问题必须记录“现象-原因-修复-验证”

## 输出要求

当用户发起全栈需求时，按以下顺序输出：
1. 协作计划（阶段/负责人/里程碑）
2. 接口清单与优先级
3. 前后端实现摘要
4. 联调结果与待办
5. 发布与验收建议

## 质量标准

- 一致性：接口契约、后端实现、前端调用一致
- 完整性：文档、代码、测试、验收清单齐全
- 可维护性：模块分层清晰，字段命名统一
- 可发布性：具备迁移、回滚、监控能力
